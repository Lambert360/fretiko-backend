# Custom JWT Authentication - Restored

## Overview
The staff authentication system has been restored to use **custom JWT tokens** generated by NestJS `JwtService` instead of Supabase Auth tokens. This removes all dependencies on the broken Supabase Admin API.

## Changes Made

### 1. StaffModule (`staff.module.ts`)
- ✅ Added `JwtModule` with async configuration
- ✅ JWT secret from `JWT_SECRET` environment variable
- ✅ Access token expires in 8 hours
- ✅ Refresh token expires in 7 days
- ✅ Exported `JwtModule` for other modules

### 2. StaffService (`staff.service.ts`)
- ✅ **Login**: Uses `JwtService.sign()` to generate custom JWT tokens
- ✅ **createStaff**: Removed Supabase Auth user creation, creates staff directly
- ✅ **changePassword**: Removed Supabase Auth password sync
- ✅ **initializeSuperAdmin**: Removed Supabase Auth user creation
- ✅ All methods now work independently of Supabase Auth

### 3. StaffJwtAuthGuard (`guards/staff-jwt-auth.guard.ts`)
- ✅ Uses `JwtService.verify()` to validate tokens
- ✅ Checks staff account exists and is active
- ✅ Attaches staff info to request object
- ✅ No Supabase Auth dependencies

## How It Works

### Login Flow
1. User provides `staffIdOrEmail` + `password`
2. System finds staff in `staff_accounts` table
3. Verifies password using `bcrypt.compare()`
4. Generates JWT tokens using `JwtService.sign()`
5. Returns `accessToken` (8h) and `refreshToken` (7d)

### Token Validation Flow
1. Guard extracts token from `Authorization: Bearer <token>` header
2. Validates token using `JwtService.verify()`
3. Extracts `payload.sub` (staff ID)
4. Verifies staff exists and is active in database
5. Attaches staff info to `request.user`

## Token Payload Structure

```typescript
{
  sub: string,           // staff_accounts.id (UUID)
  staffId: string,        // staff_accounts.staff_id (e.g., "FTK-2025-0001")
  email: string,          // staff_accounts.email
  role: string,           // staff_accounts.role
  departmentId: string,   // staff_accounts.department_id
  iat: number,            // Issued at timestamp
  exp: number             // Expiration timestamp
}
```

## Environment Variables

Make sure you have `JWT_SECRET` in your `.env` file:

```env
JWT_SECRET=your-secret-key-change-in-production
```

**Important**: Use a strong, random secret key in production!
