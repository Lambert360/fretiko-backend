# Cloud Recording & Stream Status RLS Fix

## The Problem

Cloud Recording was failing with these errors:
```
❌ Stream status update returned no rows. Stream {id} may not exist.
❌ Database update returned no data or null resourceId. Update result: []
```

The stream existed in the database, but **all updates were returning empty arrays**.

---

## Root Cause: Wrong Supabase Client

### What Was Wrong

`LiveSalesService` was using the **anon key** client:

```typescript
// ❌ WRONG: Uses SUPABASE_KEY (anon key)
import { createSupabaseClient } from '../shared/supabase.client';
this.supabase = createSupabaseClient(this.configService);
```

### Why This Failed

The `live_streams` table has **Row Level Security (RLS)** policies that require:
```sql
auth.uid() = vendor_id
```

The anon key client:
- ✅ Can SELECT rows (because it passes the user's JWT token)
- ❌ Cannot UPDATE rows (because the backend has no `auth.uid()`)

This is why:
1. Creating the stream **worked** (INSERT with user token)
2. Querying the stream **worked** (SELECT with user token)  
3. Updating status **failed** (UPDATE blocked by RLS)
4. Updating Agora IDs **failed** (UPDATE blocked by RLS)

---

## The Fix

Use the **service role key** client which bypasses RLS:

```typescript
// ✅ CORRECT: Uses SUPABASE_SERVICE_ROLE_KEY (bypasses RLS)
import { createServiceSupabaseClient } from '../shared/supabase.client';
this.supabase = createServiceSupabaseClient(this.configService);
```

### Files Changed

1. **`src/live-sales/live-sales.service.ts`**:
   - Changed import from `createSupabaseClient` → `createServiceSupabaseClient`
   - Updated constructor to use service role client
   - Added comment explaining why service role is required

---

## Why Service Role Is Required

The `LiveSalesService` performs **system-level operations** that should bypass RLS:

| Operation | Reason |
|-----------|--------|
| Update `agora_resource_id`, `agora_sid` | System fields for Cloud Recording management |
| Update `stream_url` (HLS URL) | Generated by backend polling, not user action |
| Update `stream_url` when stream ends | System operation |
| Batch updates for analytics | Background jobs with no user context |

These operations need to work even when:
- No user is authenticated in the context
- The operation is triggered by a cron job or background task
- The operation is part of an internal system process

## Lessons Learned

1. **Always use service role for system operations** - User-context clients are only for user-initiated actions
2. **RLS policies block even backend operations** - The anon key respects RLS, only service role bypasses it
3. **Empty update results indicate RLS blocking** - Not always a missing row
4. **Validate environment variables** - Ensure `SUPABASE_SERVICE_ROLE_KEY` is set in `.env`

---

## Environment Variables

Ensure these are set in `.env`:

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_KEY=eyJhbGc...  # Anon key (for user operations)
SUPABASE_SERVICE_ROLE_KEY=eyJhbGc...  # Service role key (for system operations)
```

**⚠️ IMPORTANT**: Never expose `SUPABASE_SERVICE_ROLE_KEY` to the frontend!
